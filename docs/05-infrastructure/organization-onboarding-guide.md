# Organization Onboarding Guide

## Purpose

This document defines the onboarding process for organizations entering limited rollout.

The goal is to make onboarding repeatable and clear.

## Principle

Every organization should follow the same basic onboarding path.

Avoid one-off setup that cannot be repeated later.

## Step 1: Confirm Fit

Confirm the organization:

- understands the MVP scope
- accepts known limitations
- has an owner or admin contact
- can provide feedback
- is ready to test core workflows

## Step 2: Collect Setup Details

Collect:

```text
organization_name
country
timezone
currency
default_language
owner_admin_name
support_contact
expected_initial_users
```

## Step 3: Configure Organization

Set up:

- organization profile
- owner or admin user
- starter access profile
- Religious module availability
- starter features
- Religious settings
- starter congregation, service, or group where useful

## Step 4: Verify Access

Confirm:

- owner or admin can log in
- organization appears correctly
- selected organization works
- Religious dashboard opens
- owner or admin can access expected workflows

## Step 5: Train Core Workflows

Walk through:

- login
- organization selection
- member creation
- visitor creation
- visitor conversion where available
- congregation, service, and group usage
- attendance session and manual marking
- issue reporting

## Step 6: Explain Limitations

Clearly explain what is not included yet:

- QR attendance
- GPS attendance
- live SMS
- live payment or giving
- full offline sync
- POS or Commerce
- HR or Finance

## Step 7: Confirm Support Process

Confirm the organization knows:

- support channel
- what details to include in issue reports
- expected response process
- how feedback will be reviewed

## Step 8: Record Completion

Record:

```text
organization
onboarding_date
owner_admin_confirmed
authentication_verified
core_workflows_demoed
known_limitations_shared
support_channel_shared
open_actions
```

## Final Rule

Onboarding is complete only when the organization can access the platform, understands the MVP boundaries, and knows how to request support.
