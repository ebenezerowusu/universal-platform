# Post-Launch Support Review Guide

## Purpose

This document defines how to review support activity after launch.

The goal is to identify product, training, and operations improvements from real support patterns.

## Principle

Support issues are product signals.

Repeated support questions usually mean the product, onboarding, or documentation needs improvement.

## Review Inputs

Review:

- support requests
- affected workflows
- organization context
- severity levels
- repeated user confusion
- response times
- unresolved items
- accepted limitations

## Support Categories

Group issues by:

- sign-in and access
- organization selection
- member workflow
- visitor workflow
- structure workflow
- attendance workflow
- reports or summaries
- data correction
- unavailable feature confusion
- general usability

## Review Questions

Ask:

- Which workflows generated the most support?
- Which issues repeated across organizations?
- Which issues were caused by unclear UI text?
- Which issues were caused by missing onboarding?
- Which issues were genuine product defects?
- Which requests are future features, not support problems?

## Action Classification

Classify each repeated support theme as:

```text
fix product
improve UI text
improve onboarding
improve support docs
accepted limitation
future feature
not planned
```

## Support Quality Review

Review:

- response speed
- clarity of support answers
- ability to reproduce issues
- completeness of issue reports
- handoff to development where needed

## Final Rule

The support review should reduce repeated confusion and improve core workflow success before feature expansion.
