# Post-Launch Review Template

## Purpose

This template should be copied after the first launch window or after the first launch week.

It helps the team decide what to stabilize, what to improve, and when to expand.

## Template

```text
# Post-Launch Review: <Release Name>

## Date

YYYY-MM-DD

## Release Reference

Branch, tag, commit, or build reference.

## Review Window

Start: YYYY-MM-DD
End: YYYY-MM-DD

## Summary

Short summary of launch outcome.

## Active Organizations

- organization count
- notable adoption notes

## Active Users

- user count
- usage notes

## Core Workflow Review

- sign-in
- organization selection
- Religious dashboard
- members
- visitors
- structure
- attendance

## What Worked Well

- item 1

## What Did Not Work Well

- item 1

## Support Themes

- theme 1

## Product Issues

- issue 1

## Training or Documentation Gaps

- gap 1

## Accepted Limitations

- limitation 1

## Urgent Fixes

- fix 1

## Future Opportunities

- opportunity 1

## Decision

Choose one:

- continue stabilization
- expand rollout gradually
- start next MVP feature wave
- pause expansion

## Next Actions

- action 1
```

## Final Rule

The post-launch review should be honest and evidence-based. Do not treat launch as successful without reviewing actual usage and support patterns.
