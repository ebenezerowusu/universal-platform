# Payment Engine Specification

## Purpose

The Payment Engine provides one platform-wide way to initiate, verify, record, reconcile, and report payments across all domains.

Domains must not integrate directly with payment providers.

## Engine Position

```text
Domain Module
  -> Payment Engine
    -> Payment Provider Resolver
      -> Provider Adapter
        -> External Payment Provider
```

## Responsibilities

The Payment Engine is responsible for:

- Payment initialization
- Payment verification
- Webhook/callback processing
- Transaction recording
- Payment status tracking
- Digital receipts
- Provider abstraction
- Country/provider resolution
- Organization payment configuration
- Payment audit logs
- Payment events
- Payment reporting foundation

## Non-Responsibilities

The Payment Engine does not decide religious giving rules, marketplace order rules, HR payroll rules, or subscription plan business rules.

Those belong to their domains or platform modules.

The Payment Engine only provides payment capability.

## Domains That Use It

Examples:

- Religious Domain: tithes, offerings, donations, event payments, pledges
- Commerce Domain: checkout, orders, delivery fees
- Subscription Module: organization subscription billing
- Events Module: ticket payments
- Future domains: any paid activity

## Provider Adapters

Potential adapters:

- Hubtel
- Paystack
- Flutterwave
- M-Pesa
- PayFast
- Bank transfer adapter
- Manual payment adapter
- Future providers

Adapters must implement a common provider interface.

## Provider Resolution

Payment provider selection should be configuration-driven.

Possible resolution inputs:

- Organization country
- Currency
- Payment method
- Organization settings
- Platform provider configuration
- Fallback provider rules

Example:

```text
Country: Ghana
Currency: GHS
Method: mobile_money
Resolved Provider: Hubtel or Paystack based on platform configuration
```

The organization user should not need to know the low-level provider implementation.

## Payment Lifecycle

Common lifecycle:

```text
created
initialized
pending
successful
failed
cancelled
expired
refunded
reversed
```

## Core Payment Flow

```text
Domain requests payment
  -> Payment Engine validates request
  -> Payment Engine creates transaction
  -> Provider Resolver chooses provider
  -> Provider Adapter initializes payment
  -> User completes payment
  -> Provider sends callback/webhook
  -> Payment Engine verifies provider response
  -> Payment status updated
  -> PaymentCompleted or PaymentFailed event published
  -> Domain reacts to event
```

## Domain Metadata

The Payment Engine should allow domain-specific metadata without understanding domain internals.

Example metadata:

```json
{
  "religious_giving_type": "tithe",
  "branch_id": "...",
  "pledge_id": "..."
}
```

The Payment Engine stores metadata but should not enforce religious giving rules.

## Security Requirements

- Webhooks must be verified.
- Provider secrets must never be exposed to clients.
- Payment callbacks must be idempotent.
- Duplicate provider events must not create duplicate successful payments.
- All payment state changes must be auditable.
- Payment amounts must be validated server-side.
- The client must not be trusted for final payment status.

## Idempotency

Payment initialization and webhook handling must support idempotency.

Use idempotency keys and provider event references where possible.

## Tenant Isolation

Every payment transaction must belong to an organization unless it is a platform-level transaction.

Organization users must not see another organization's payment records.

## Receipts

Receipts should be generated by the Payment Engine or through a Receipt service using Payment Engine data.

Domains may provide display labels, but receipt integrity must come from verified payment records.

## Testing Requirements

Tests must cover:

- Payment initialization
- Provider resolution
- Successful callback
- Failed callback
- Duplicate callback handling
- Invalid webhook signature
- Tenant isolation
- Permission checks
- Provider adapter mocking

## Final Rule

Domains request payments. The Payment Engine owns payment processing.
