# Sprint 32 Notification Center Roadmap

## Purpose

This document defines Sprint 32 for Notification Center and User Preferences foundation.

Sprint 32 should create a reusable notification layer for in-app messages, delivery logs, templates, preferences, and domain trigger references.

## Sprint Goal

Enable organizations and users to receive, review, manage, and configure safe notifications generated by platform domains while keeping provider-specific delivery behind Notification, SMS, and future communication engines.

## Why This Sprint

Many modules now need notifications:

- calendar reminders
- order status updates
- delivery status updates
- fulfillment updates
- leave approvals
- attendance alerts
- finance/reconciliation alerts
- religious event reminders
- system/admin notices

Without a shared notification center, each module will create separate messaging logic.

## Work Package 1: In-App Notification Records

Prepare notification records with:

- recipient user optional
- organization
- title
- body/safe summary
- type/category
- status
- read timestamp optional
- archived timestamp optional

## Work Package 2: Notification Types and Categories

Prepare categories such as:

```text
system
reminder
approval
order_update
delivery_update
finance_alert
attendance_alert
religious_event
custom
```

## Work Package 3: Template Metadata

Prepare template metadata foundation without building a full marketing template engine.

Templates should be provider-agnostic and safe for internal system notifications.

## Work Package 4: User Preferences

Prepare user preference foundation:

- channel preference placeholder
- category preference
- muted category placeholder
- quiet hours placeholder
- status

## Work Package 5: Organization Preferences

Prepare organization-level defaults where practical.

User preferences should override organization defaults where policy allows.

## Work Package 6: Delivery Logs

Prepare delivery logs for:

- in-app created
- SMS requested placeholder
- email requested placeholder
- failed
- skipped due to preference

## Work Package 7: Domain Trigger References

Prepare domain trigger references such as:

```text
calendar_reminder
order_status_change
delivery_status_change
leave_request_update
attendance_alert
finance_alert
religious_event_reminder
custom
```

## Work Package 8: Flutter Foundation

Prepare screens or placeholders for:

- notification inbox
- notification detail
- unread filter
- archive action
- preferences
- delivery warning placeholder

## Out of Scope

Do not implement:

- full marketing campaign engine
- push provider integration
- email provider integration
- WhatsApp provider integration
- advanced personalization
- unrestricted broadcast messaging
- bulk SMS sales expansion

## Completion Standard

Sprint 32 is complete when users can receive in-app notifications, mark them read, archive them, configure basic preferences, and domains can link notifications to trigger references through provider-agnostic APIs.

## Final Rule

Notification Center must centralize user-facing messages while keeping delivery providers behind engines.
