# Sprint 58 Knowledge Base Help Center Self Service Roadmap

## Purpose

This document defines Sprint 58 for Knowledge Base, Help Center, and Self-Service Support foundation.

Sprint 58 should create a reusable help content layer for articles, article versions, categories, collections, visibility rules, search linkage, support escalation links, feedback, localization placeholders, related policy/training links, and audit history.

## Sprint Goal

Enable organizations and platform operators to publish searchable help content, guide users through common tasks, collect feedback, and escalate unresolved help needs to support workflows.

## Why This Sprint

The platform now has many operational surfaces that need guided self-service:

- organization onboarding
- policy acknowledgements and training
- support operations
- document/media library
- search and discovery
- workflow approvals
- data portability and privacy reviews
- finance, HR, religious, commerce, and logistics modules

Without a shared knowledge base foundation, each module may create isolated help pages, support notes, or static documentation with inconsistent visibility and no feedback loop.

## Work Package 1: Help Article Metadata

Prepare article records with:

- article key
- organization scope optional
- owner module/domain
- title
- category
- status

## Work Package 2: Article Version Placeholder

Prepare article version placeholders for:

- version label
- content reference placeholder
- document/media reference placeholder
- published date placeholder
- replaced version placeholder

## Work Package 3: Categories and Collections

Prepare category and collection records for:

- user guides
- admin guides
- troubleshooting
- onboarding
- policy/training references
- module-specific help

## Work Package 4: Visibility and Audience Rules

Prepare visibility placeholders for:

- platform-public placeholder
- organization-only
- role-based
- module-enabled tenants
- branch/network scoped placeholder

## Work Package 5: Search Linkage

Prepare search indexing linkage placeholders so articles can be discoverable through Search/Discovery.

## Work Package 6: Support Escalation Linkage

Prepare escalation placeholders so users can create or link support cases when an article does not resolve their issue.

## Work Package 7: Article Feedback

Prepare feedback records for:

- useful/not useful
- issue still unresolved placeholder
- suggested improvement placeholder
- content outdated placeholder

## Work Package 8: Related Policy and Training Links

Prepare related content links to policy library, training assignments, risk controls, and document/media references.

## Work Package 9: Flutter Foundation

Prepare screens or placeholders for:

- help center dashboard
- article list
- article detail
- category/collection list
- article version detail
- search results placeholder
- feedback screen
- support escalation placeholder
- help content audit history

## Out of Scope

Do not implement:

- full CMS replacement
- AI support bot as source of truth
- unrestricted public help publishing
- bypass of document or policy permissions
- cross-organization private article visibility
- arbitrary script widgets in help articles
- replacement of Support Operations

## Completion Standard

Sprint 58 is complete when help articles, versions, categories, collections, visibility rules, search linkage, escalation links, feedback, related content links, and audit history are available through permission-protected and auditable foundations.

## Final Rule

Knowledge base content should reduce support load while preserving visibility rules, ownership, and escalation paths.
